一、IOC/DI简介

1.什么是IOC容器？

  控制反转即IOC，它把传统上由程序代码直接操作对象的调用权交给容器，通过容器来实现对象组件的装配和管理。所谓的控制反转概念就是组件对控制权的转移，从代码本身转移到了外部容器。

2.IOC的优点

  • IOC或者依赖注入把应用的代码量降到了最低
  • 它使应用容易测试，单元测试不需要单例和JNDI查找机制
  • 最小的代价和最小的侵入式使松散耦合得以实现
  • IOC容器支持加载服务时的饿汉式初始化和懒加载

3.IOC的实现机制

  Spring的IOC实现原理就是工厂模式加反射机制。

4.依赖注入(DI)

  依赖注入和控制反转时对同一事件的不同描述。依赖注入是从程序的角度，应用程序依赖IOC容器创建并注入需要的外部资源；控制反转则是从IOC的角度，IOC容器控制应用程序，由容器反向注入程序需要的外部资源。

5.依赖注入的基本原则

  应用组件不应该负责查找资源或者其他依赖的协助对象。配置对象的工作应该由IOC容器负责，”查找对象”的逻辑应该从应用组件的代码中抽取出来，交给IOC容器负责，容器全权负责组件的装配，他会把符合依赖关系的对象通过属性或者构造器传递给需要的对象

6.依赖注入的优势

  • 查找定位操作与应用代码完全无关
  • 不依赖容器的API，可以容易的再任何容器意外使用应用对象
  • 不需要特殊的接口，绝大数对象可以做到完全不必依赖容器

7.IOC/DI的实现

常用方法：

  • getBean(String name)

    • 可以根据Bean的id生成Bean的id和相对应的类生成Bean实例

  • getBean(String name Class requiredType)

    • 可以根据Bean的id和相对应的类生成Bean实例

常用实现类：

  • FileSystemXmlApplicationContext

    • 从文件系统中的Xml文件获取上下文中定义的信息

  • ClassPathXmlApplicationContext

    • 从类路径中的Xml文件获取上下文中定义的信息

  • XmlWebApplicationContext

    • 从Web服务器中的Xml文件加载上下文中定义的信息

ClassPathXmlApplicationContext与FileSystemXmlApplicationContex的区别

  • FileSystemXmlApplicationContext：只能在指定的路径中查询配置文件
  • ClassPathXmlApplicationContext：可以在整个类路径中查询配置文件

8.BeanFactory与ApplicationContext的区别

  BeanFactory和Application是Spring的两大核心接口，都可以当作Spring的容器，其中Application是BeanFactory的子接口。

  Application接口作为BeanFactory的派生，除了提供BeanFactory所具有的功能外，还提供了更完整的框架功能：

  • 继承MessageSource支持国际化
  • 统一资源文件访问方式
  • 提供在监听器中注册Bean的时间
  • 同时加载多个配置文件
  • 载入多个上下文，使每个上下文都专注于一个特定的层次

加载方式

  • BeanFactory采用的延时加载形式在注入Bean的，即使只有使用到某个Bean时(getBean())才对该Bean进行加载实例化，这样就不能发现一些存在Spring的配置问题，如果Bean的某一个属性没有注入，BeanFactory加载后，直至第一次调用getBean方法才会抛出异常
  • ApplicationContext它是容器启动时一次性创建了所有的Bean，这样容器启动时我们就可以发现Spring中存在的配置错误，有利于检查所依赖的属性是否注入，ApplicationContext启动后预载入所有的单例Bean，通过预载入单实例Bean，确保当你需要的时候不用等待。
  • 相对于基本的BeanFoctory，ApplicationContext唯一的不足是占用内存空间。

创建方式

  BeanFactory通常以编程的方式被创建，而ApplicationContext不仅可以以编程的方式别创建还能以声明的方式创建，如使用ContextLoader。

注册方式

  BeanFactory和ApplicationContext都支持BeanPostProcessor、BeanFactoryPostPR、rocessor的使用。但BeanFactory需要手动注册，而ApplicationContext则是自动注册

四、Bean的作用域

Spring框架一共支持五种Bean的作用域

singleton：Bean在每个ioc容器中只有一个实例，单例模式。

prototype:一个Bean的定义可以有多个实例，原型模式。

request：每次http请求都会创建一个Bean，该作用域尽在基于Web的Spring。

session：在一个HTTP Session中，一个Bean定义对应一个实例。该作用域仅在基于Web的Spring ApplicationContext情形下有效。

global-session：在一个全局的HTTP Session中，一个Bean定义对应一个实例。该作用域尽在基于Web的Spring ApplicationContext

注意：

  默认的Spring Bean的作用域是Singleton(单例模式)。使用prototype作用域需要慎重思考，因为频繁的创建和销毁Bean会带来很大的性能开销。

通过Bean标签中的 scope属性可以设置Bean的作用域

五、使用注解装配

使用注解开发Spring配置文件中要包含<context:annotation-config>标签来开启注解功能

@Component : 将这个类标记为Spring Bean，它是任何Spring管理主键的通用类型.

@Repository ：将数据库访问层(DAO)的类标记为Spring Bean，与@Component功能相同。

@Service : 将业务层(service)的类标明为Spring Bean，与@Component功能相同。

@Controller : 将控制层(Controller)的类标明为Spring Bean ，与@Component功能相同。

@Autowired : 默认是按照属性类型装配注入的，默认情况下它要求依赖对象必须存在(可以设置它的required属性为false)。

@Resource : 与@Autowired作用一样，区别在于@Autowired是按照Bean类型装配，@Resource是按照Bean实例名称进行装配，@Resource有两个重要属性：name和type，Spring把name属性解析成Bean的名称，type解析成Bean的实例类型

@Qualifier : 与@Autowired 配合使用，会将Bean的默认类型按照Bean的实例名称装配

@Value : 给基本数据类型赋值

@Scope : 设置Bean的作用域

@Nullable : 被此注解标记的对象可以为空

  接下来我们来看一下使用注解后的开发

DAO层

自动装配

Spring的bean包含一个autowired属性，我们可以通过设置autowired设置bean的自动装配

  • default(默认值)：由Bean的上级标签Beans来确定，设置Beans所有Bean的装配模式
  • byName：根据名称自动装配
  • byType：更近数据类型自动装配
