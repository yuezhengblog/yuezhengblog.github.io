JWT

一、简介

1.1 什么是JWT？

​ JSON Web令牌(JWT)是一个开放标准(RFC 7519),它定义了一种尽愁且自包含的方式，用于在各方之间安全地将信息作为JSON对象传输。由于此信息是经过数字签名地，因此可以被验证和信任。可以使用秘密(使用HMAC算法)或使用RSA或ECDSA的公钥/私钥对进行签名

你在说啥？什么玩意？

​ 简单通俗的解释就是：JWT也就是通过JSON形式作为Web应用中的令牌，用于在各方之间安全的将信息作为JSON对象传输。在数据传输过程中还可以完成数据加密、签名等相关处理！

1.2 JTW能做什么？

授权：

​ 这时使用JWT最常见的方案。一旦用户登陆，每个后续请求将包括JWT、从而允许用户访问令牌允许的路由、服务和资源。单点登陆是当今广泛使用JWT的一项功能，因为它的开销很小而且可以在不同的域中轻松使用。

信息交换：

​ JTW是各方之间安全地传输信息地好方法。因为可以对JWT进行签名，所以您可以确保发件人是他们说所得人。此外，由于签名是使用表头和有效负载计算的，因此您还可以验证内容是否遭到篡改。

1.3 JWT使用环境

第一种：JavaWeb种的安全验证

​ 前端和后端的数据访问，前端访问后端携带JTW令牌，后端验证令牌，如果成功返回数据，如果失败拒绝访问！

第二种：信息校验

​ 在系统之间使用令牌机制用来信息的传递，例如A系统于B系统，使用JWT来信息传递并在传递过程中还可以对数据进行签名，A系统将发送的数据进行签名，B系统拿到数据后进行解析，如果解析后的数据于A系统的数据一致，就可以证明数据传输种没有被篡改。

1.4 JWT优点

基于传统的Session认证

  1. 认证方式

    ​ 我们指定，http协议本身是一种无状态的协议，而这就意味着如果用户向我们的应用提供了用户名和密码来进行用户认证，那么下一次请求，用户还要再一次进行用户认证才行，因为根据http协议，我们并不知道是哪个用户发出的请求，所以为了我们的应用能识别是那个用户发出的请求，我们只能在服务器存储一份用户登陆的信息，这份信息会在影响时传递给浏览器，告诉其保存为cookie，以便下次请求时发送给我们的应用，这样我们的应用就能识别请求来自那个用户了，这就是传统的基于Session认证。

  2. 认证流程

  3. 暴露问题

    • 每个用户经过我们的应用都要在服务器做一次记录，以便用户下次请求的鉴别，通常而言session都是保存在内存中的，而随着认证用户的增多，服务器的开销会明显增大

    • 用户认证之后，服务端做认证记录，如果认证的记录被保存在内存中的话，这意味着用户下次请求必须要请求在存放它session的服务器上，这样才能拿到授权的资源，这样在分布式的应用上，相应的限制了负载均衡器的能力。这也就意味着限制了应用的扩展能力

    • 因为是基于cookie来进行用户的识别，cookie如果被截获，用户就会很容易受到跨站请求伪造的攻击

    • 在前后端分离系统中就更加痛苦：

      也就是说前后端分离在应用解耦后增加了部署的复杂性。通常用户一次请求就要转发多次。如果session每次携带sessionid到服务器，服务器还要查询用户信息。同时如果用户很多。这些信息存储在服务器内存中，给服务器增加负担。还有就是CSRF(跨站伪造请求攻击)攻击，session是基于cookie进行用户识别的,cookie如果被截获，用户就会容易受到跨站请求伪造的攻击。还有就是sessionid就是一个特征值，表达的信息不够丰富。不容易扩展。而且后端应用是多部署。那么就需要实现session共享机制。不方便集群应用

基于JWT认证

认证流程：

  1. 首先，前端通过Web表单将自己的用户名和密码发送到后端的接口。这一过程一般是一个HTTP POST请求。建议的方式是通过SSL加密的传输(https协议)，从而避免敏感信息被嗅探。
  2. 后端核对用户名和密码成功后，将用户的id等其他信息作为JWT Payload(负载)，将其与头部发呢别进行Base64编码拼接后签名，形成一个JWT(Token)。形成的JWT就是一个形同lll.zzz.xxx的字符串；Token:head.payload.signature
  3. 后端将JWT字符串作为登陆成功的返回结果返回给前端。前端可以将返回的结果保存在localStorage或者sessionStorage上，退出登陆时前端删除保存的JWT即可。
  4. 前端在每次请求时将JWT放入HTTP Header中的Authorization位。(解决XSS和XSRF问题)
  5. 后端检查是否存在，如存在验证JWT的有效性。例如，检查签名是否正确、检查Token是否过期、检查Token的接收方是否是自己(可选)。
  6. 验证通过后后端使用JWT中包含的用户信息进行其他逻辑操作，返回响应结果

JWT优势：

  • 简洁：
    • 可以通过URL、POST参数或者在HTTP header发送，因为数据量小，传输速度也很快
  • 自包含
    • 负载中包含了所有用户所需要的信息，避免了多次查询数据库
  • 跨语言
    • 因为Token是以JSON加密的形式保存在客户端的，所以JWT是跨语言的，原则上任何web形式都支持。
  • 分布式
    • 不需要再服务端保存会话信息，特别使用于分布式微服务

二、JWT 结构

2.1 令牌组成(Token)

​ Token实际上就是一个xxx.yyy.zzz样式的字符串，分别对应着：标头(Header)、有效载荷(Payload)、签名(Signature)；因此它的类型就是Header.Payload.Signature

2.2 Header

​ 表头通常由两部分组成：令牌的类型(即JWT)和所使用的签名算法，例如：HMAC、SHA256或RSA。它会使用Base64编码组成JWT结构的一部分。

注意：Base64是一种编码，也就是说，它是可以被翻译回原来的样子的。它并不是一种加密过程

​ 这里它用的是typ而不是type估计也是少一个字符少可以少占用一点空间吧。而HS256则是JWT默认也是推荐的算法

2.3 Payload

​ 令牌的第二部分是有效负载，其中包含声明。声明是由关实体(通常是用户)和其他数据的声明。同样的，它会使用Base64编码组成JWT结构的第二部分。Token它是一个自包含，何为自包含呢？就是在token信息中包含着它自身想要的用户信息，它自身想要的用户信息就是放在Payload中。但是不要把敏感信息(如密码)放在Payload，这也是官方建议的。

2.4 Signature

​ 前面两个部分都是使用Base64进行编码的，即前端可以解开知道里面的信息。Signature需要使用编码后的Header和Payload以及我们提供的一个密钥，然后使用header中指定的前面算法进行签名。签名的作用是保证JWT没被篡改过。

2.5 签名的目的

​ 最后一步签名的过程，实际上是对头部以及负载内容进行签名，放置内容被篡改。如果有人对头部以及负载的内容节码之后进行修改，再进行编码，最后加上之前的签名组成的新的JWT的话，那么服务器会判断出新的头部和负载形成的签名和JWT附带上的签名是不一样的。如果要对新的头部和负载进行签名，再不知道服务器加密时用的密钥的话，得出来的签名也是不一样的。
